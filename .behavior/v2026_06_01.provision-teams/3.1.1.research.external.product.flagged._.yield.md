# research: flagged topics from vision

## topic.1 = delete team with children (API behavior)

### findings

[FACT] if you are an organization owner, deleting a parent team will delete all of its child teams as well [1][6]

[FACT] only organization owners can delete parent teams; team maintainers cannot delete teams that have child teams [1]

[FACT] the deletion cascades automatically — no manual step required to remove children first [1][6]

### citations

1. [Deleting a team - GitHub Docs](https://docs.github.com/en/organizations/organizing-members-into-teams/deleting-a-team) [official] 2024
   > "If you are an organization owner, deleting a parent team will delete all of its child teams as well."

6. [REST API endpoints for teams - GitHub Docs](https://docs.github.com/en/rest/teams/teams) [official] 2024
   > "To delete a team, the authenticated user must be an organization owner or team maintainer."

### convergence

sources [1] and [6] agree: child teams are deleted when parent is deleted.

## topic.2 = remove last maintainer (API behavior)

### findings

[KHUE] can a team have zero maintainers? documentation is unclear

[SUMP] github recommends "at least one maintainer" but does not document a hard constraint [2][3]

[FACT] for organization owners, github requires "no less than two people" — but this is org-level, not team-level [2]

[OPIN] based on the absence of documented constraint, the API likely allows removal of last maintainer, leaving a team with only members

### citations

2. [Roles in an organization - GitHub Docs](https://docs.github.com/en/organizations/managing-peoples-access-to-your-organization-with-roles/roles-in-an-organization) [official] 2024
   > "The role should be limited, but to no less than two people, in your organization."

3. [Assigning the team maintainer role to a team member - GitHub Docs](https://docs.github.com/en/organizations/organizing-members-into-teams/assigning-the-team-maintainer-role-to-a-team-member) [official] 2024
   > "It is often useful for a team to have at least one maintainer."

### conflict

no sources explicitly document whether removal of last maintainer is blocked. sources [2][3] suggest it is recommended but not required. requires empirical test.

## topic.3 = team rate limits

### findings

[FACT] GitHub Apps on Enterprise Cloud: 15,000 requests/hour [4]

[FACT] GitHub Apps on non-Enterprise: 5,000-12,500 requests/hour (scales with repos/users) [4]

[FACT] OAuth apps: 5,000 requests/hour (or 15,000 for Enterprise Cloud orgs) [4]

[FACT] secondary rate limits apply: max 100 concurrent requests, per-endpoint limits per minute [4]

[SUMP] team operations use standard rate limits — no team-specific limits documented [4]

### citations

4. [Rate limits for the REST API - GitHub Docs](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api) [official] 2024
   > "For GitHub Apps installed on a GitHub Enterprise Cloud organization, the rate limit is 15,000 requests per hour."
   > "Secondary rate limits prevent abuse and are triggered by making more than 100 concurrent requests."

5. [Enterprise teams product limits increased - GitHub Changelog](https://github.blog/changelog/2025-12-08-enterprise-teams-product-limits-increased-by-over-10x/) [official] 2025
   > "Enterprise teams remain assignable across up to 1,000 organizations per enterprise."

### convergence

sources [4][5] agree on standard rate limit structure. no team-specific limits found.

## topic.4 = declastruct dependency order

### findings

[FACT] terraform determines resource order via dependency graph, not declaration order [7][8]

[FACT] implicit dependencies are inferred from resource attribute references [7][8][9]

[FACT] when resource A references resource B, terraform creates B before A [7]

[SUMP] declastruct likely uses similar ref-based dependency ordering since it follows terraform patterns

[KHUE] does declastruct support explicit depends_on for edge cases?

### citations

7. [Create resource dependencies | Terraform - HashiCorp](https://developer.hashicorp.com/terraform/tutorials/configuration-language/dependencies) [official] 2024
   > "Terraform uses dependency information to determine the correct order in which to create the different resources. To do so, it creates a dependency graph."

8. [How to Manage Dependencies Between Terraform Resources - Spacelift](https://spacelift.io/blog/terraform-dependencies) [practitioner] 2024
   > "The order in which resources are declared in your configuration files has no effect on the order in which Terraform creates or destroys them."

9. [Terraform - Understanding Implicit and Explicit Dependencies - DEV](https://dev.to/pwd9000/terraform-understanding-implicit-and-explicit-dependencies-n9l) [blog] 2024
   > "Terraform automatically infers when one resource depends on another by studying the resource attributes used in interpolation expressions."

### convergence

sources [7][8][9] strongly agree: refs create implicit dependencies that control order. this is the standard IaC pattern.
