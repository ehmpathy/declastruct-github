# blocker.1: SDK operation unit tests lack exhaustive snapshot coverage for contracts

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.verification/rule.require.contract-snapshot-exhaustiveness.md

**locations**:
- src/domain.operations/team/delTeam.test.ts
- src/domain.operations/team/getOneTeam.test.ts
- src/domain.operations/team/setTeam.test.ts
- src/domain.operations/teamMembership/delTeamMembership.test.ts
- src/domain.operations/teamMembership/getOneTeamMembership.test.ts
- src/domain.operations/teamMembership/setTeamMembership.test.ts

Multiple SDK operation test files (getOneTeam, setTeam, delTeam, and equivalents for teamMembership) only contain a minimal 'function exports' test that verifies the function is available via typeof check. There are no calls exercising the methods, no positive path (e.g. found resource), negative path (e.g. not-found), edge cases (e.g. validation errors, idempotency), and crucially no toMatchSnapshot calls on the contract outputs (return values for SDK). This directly matches the negative example in the rule of 'tests verify the behavior works' with 'no snapshots, no error cases, no edge cases'. Although play.integration.test.ts files (also in target) contain some snapshots, the rule requires exhaustive snapshot coverage for every SDK method's output variants in the tests for the contract. These files create blind spots for caller experience, contract drift, and regressions. Need to add snapshots for all variants (positive, negative, edge) directly exercising the SDK methods, using stable/normalized snapshots like in the acceptance and play tests.

**snippet**:
```ts
describe('getOneTeam', () => {
  given('[case1] function exports', () => {
    when('[t0] imported', () => {
      then('it should be available', () => {
        expect(typeof getOneTeam).toBe('function');
      });
    });
  });

  // lifecycle tests for error cases and edge cases
  // are covered in team.play.integration.test.ts
  // to avoid team pollution between test files
});
```

# blocker.2: Acceptance tests verify CLI/SDK contract outputs without snapshots for variants

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.verification/rule.require.contract-snapshot-exhaustiveness.md

**locations**:
- src/contract/sdks/declastruct.acceptance.test.ts
- src/contract/sdks/declastruct.acceptance.test.ts
- src/contract/sdks/declastruct.acceptance.test.ts
- src/contract/sdks/declastruct.acceptance.test.ts
- src/contract/sdks/declastruct.acceptance.test.ts

In src/contract/sdks/declastruct.acceptance.test.ts (the primary contract test file), several 'then' blocks for user-faced contracts (CLI plan/apply commands and SDK get returns) perform verification via direct expects on outputs but omit snapshots. Examples: 'plan includes repo, config...' checks presence/slugs with expects but no snapshot of the plan resources output variant; 'plan file is created' checks existence/structure with expects but no snapshot; idempotent replan checks (environment KEEP, team KEEP, membership KEEP) assert action==='KEEP' on parsed plan but only snapshot stdout in one case and no snapshot of the plan changes JSON for the idempotent variant. This leaves gaps for positive/negative/edge variants of CLI stdout, plan output, and SDK returns. Violates 'every variant that a caller could encounter must be snapped' and 'gaps in coverage are blind spots'. Need to add toMatchSnapshot for these output variants (e.g. replan plan changes, resource summary) in addition to existing ones.

**snippet**:
```ts
then('it is idempotent - replan after apply shows team as KEEP', () => {
  const replanFile = join(testDir, 'replan.json');
  const replan = JSON.parse(readFileSync(replanFile, 'utf8'));
  const teamChanges = replan.changes.filter(
    (c: DeclastructChange) =>
      c.forResource.class === 'DeclaredGithubTeam',
  );
  expect(teamChanges.length).toBeGreaterThan(0);
  for (const teamChange of teamChanges) {
    expect(teamChange.action).toBe('KEEP');
  }
});
```
