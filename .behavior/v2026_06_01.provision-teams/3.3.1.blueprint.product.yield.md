# blueprint: provision github teams

## summary

add declarative team provision to declastruct-github:
- DeclaredGithubTeam — team creation, update, deletion, nested hierarchy
- DeclaredGithubTeamMembership — member onboard, offboard, role changes

follows extant patterns:
- DomainEntity with primary/unique/readonly fields
- DeclastructDao with get.one.byUnique, set.findsert, set.upsert, set.delete
- RefByUnique for parent team and org references
- given/when/then tests with snapshots

## notes

### dependency order

per `3.1.1.research.external.product.flagged._.yield.md` topic.4 [7][8][9]:
> "Terraform uses dependency information to determine the correct order in which to create the different resources. To do so, it creates a dependency graph."

declastruct orders resources by reference graph:
1. DeclaredGithubOrg (already exists)
2. DeclaredGithubTeam (depends on org)
3. DeclaredGithubTeam with parent (depends on parent team)
4. DeclaredGithubTeamMembership (depends on team)

### delete cascade

per `3.1.1.research.external.product.flagged._.yield.md` topic.1 [1][6]:
> "If you are an organization owner, deleting a parent team will delete all of its child teams as well."

declastruct handles via dependency graph — children deleted first.

### delete with members

per `3.1.1.research.external.product.flagged._.yield.md` topic.5:
no documented requirement to remove members before delete — deletion handles members automatically.

### last maintainer

per `3.1.1.research.external.product.flagged._.yield.md` topic.2:
[KHUE] unclear if API blocks removal of last maintainer. documentation suggests "at least one maintainer" is recommended but not required. validate empirically in integration tests.

### acceptance test resource declarations

concrete example for `resources.acceptance.ts`:

```typescript
const team = DeclaredGithubTeam.as({
  org: { login: 'ehmpathy' },
  slug: `acceptance-test-team-${testRunId}`,
  name: 'Acceptance Test Team',
  description: 'test team for declastruct acceptance',
  privacy: 'closed',
  notifications: 'disabled',

const childTeam = DeclaredGithubTeam.as({
  org: { login: 'ehmpathy' },
  slug: `acceptance-test-child-team-${testRunId}`,
  name: 'Acceptance Test Child Team',
  description: 'test child team for nested hierarchy',
  privacy: 'closed',
  notifications: 'disabled',
  parent: { org: { login: 'ehmpathy' }, slug: team.slug },

const membership = DeclaredGithubTeamMembership.as({
  team: { org: { login: 'ehmpathy' }, slug: team.slug },

return [repo, repoConfig, environment, team, childTeam, membership];
```

note: org reference uses inline `{ login: 'ehmpathy' }` — no DeclaredGithubOrg resource needed in return array since orgs are extant infrastructure.

### acceptance test org

use ehmpathy org for acceptance tests:
- org owner permissions required for team operations
- create test teams with unique names per run
- cleanup at end of test

### operation name convention

uses `getOneTeam`, `getOneMembership` (newer `getOne*` pattern, same as `getOneOrg`, `getOneOrgSecret`, `getOneOrgVariable`). extant codebase has both patterns — `getOne*` (newer) and `get*` (older like `getEnvironment`, `getRepo`). blueprint follows newer pattern for consistency with recent additions.

### delete idempotency

`delTeam` and `delMembership` are idempotent — no error if resource not found:

```typescript
// idempotent: no error if not found
if (error.message.includes('Not Found')) return;
```

follows extant `delEnvironment.ts:65-66` pattern.

### error messages

follow extant patterns:
- `team '${slug}' not found in org '${org}'`
- `secret teams cannot have a parent`
- `user '${username}' is not a member of org '${org}'`
