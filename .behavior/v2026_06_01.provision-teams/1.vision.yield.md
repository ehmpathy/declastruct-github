# vision: provision github teams

## the outcome world

### day-in-the-life with this in place

a platform engineer declares their org's team structure in code:

```typescript
const teamEngineers = DeclaredGithubTeam.as({
  org: { login: 'acme-corp' },
  slug: 'engineers',
  name: 'Engineers',
  description: 'core engineers team',
  privacy: 'closed',
  notifications: 'enabled',
  parent: null,
});

const teamFrontend = DeclaredGithubTeam.as({
  org: { login: 'acme-corp' },
  slug: 'frontend',
  name: 'Frontend',
  description: 'frontend specialists',
  privacy: 'closed',
  notifications: 'enabled',
  parent: { org: { login: 'acme-corp' }, slug: 'engineers' },
});

const teamFrontendMembers = ['alice', 'bob', 'charlie'].map((username) =>
  DeclaredGithubTeamMembership.as({
    team: { org: { login: 'acme-corp' }, slug: 'frontend' },
    username,
    role: 'member',
  }),
);
```

they run `declastruct plan` to see what will change, then `declastruct apply` to converge.

### before/after contrast

| before | after |
|--------|-------|
| click through github UI for each team | declare teams in code |
| manual member adds, easy to miss | membership as code, auditable |
| drift between orgs, no single source of truth | terraform-like state management |
| onboard = manual checklist | onboard = add username to config |
| offboard = hope we removed them everywhere | offboard = remove from config, apply |

### the "aha" moment

> "wait, i can see every team and member in a single file, and git blame shows who changed what and when? and CI can validate changes before they're applied?"

the value clicks when:
1. new hire onboard becomes a PR that adds their username to the right teams
2. offboard becomes a PR that removes their entries then runs apply
3. team restructure becomes a refactor with review and rollback capability

## user experience

### usecases and goals

| usecase | goal |
|---------|------|
| org bootstrap | create initial team structure for new org |
| team creation | add a new team with specific privacy/notification settings |
| nested teams | create child teams under parent teams |
| member onboard | add user to one or more teams with correct roles |
| member offboard | remove user from all teams |
| role changes | promote member to maintainer or demote |
| team restructure | move teams under different parents |
| audit | see who is in what team and when changes occurred |

### contract inputs & outputs

#### DeclaredGithubTeam

```typescript
interface DeclaredGithubTeam {
  // metadata (read-only, set by github)
  id?: number;
  createdAt?: UniDateTime;
  updatedAt?: UniDateTime;

  // unique key
  org: RefByUnique<typeof DeclaredGithubOrg>;
  slug: string;

  // team properties
  name: string;
  description: string | null;
  privacy: 'secret' | 'closed';
  notifications: 'enabled' | 'disabled';

  // hierarchy (null = top-level team)
  parent: RefByUnique<typeof DeclaredGithubTeam> | null;
}
```

#### DeclaredGithubTeamMembership

```typescript
interface DeclaredGithubTeamMembership {
  // unique key (composite)
  team: RefByUnique<typeof DeclaredGithubTeam>;
  username: string;

  // membership properties
  role: 'member' | 'maintainer';
}
```

### what it looks like to leverage them

```typescript
// resources.ts - full org team structure
export const getResources = async () => {
  const org = DeclaredGithubOrg.as({ login: 'acme-corp', ... });

  // top-level teams
  const teamEngineers = DeclaredGithubTeam.as({
    org,
    slug: 'engineers',
    name: 'Engineers',
    description: 'all engineers',
    privacy: 'closed',
    notifications: 'enabled',
    parent: null,
  });

  const teamOperations = DeclaredGithubTeam.as({
    org,
    slug: 'operations',
    name: 'Operations',
    description: 'devops and sre',
    privacy: 'closed',
    notifications: 'enabled',
    parent: null,
  });

  // nested team
  const teamFrontend = DeclaredGithubTeam.as({
    org,
    slug: 'frontend',
    name: 'Frontend',
    description: 'ui/ux engineers',
    privacy: 'closed',
    notifications: 'enabled',
    parent: teamEngineers,
  });

  // memberships
  const teamFrontendMembers = ['alice', 'bob'].map((username, i) =>
    DeclaredGithubTeamMembership.as({
      team: teamFrontend,
      username,
      role: i === 0 ? 'maintainer' : 'member',
    }),
  );

  return [org, teamEngineers, teamOperations, teamFrontend, ...teamFrontendMembers];
};
```

### timelines

1. **plan** - user runs `declastruct plan`, sees diff of current vs desired state
2. **review** - team reviews the plan (especially membership changes)
3. **apply** - user runs `declastruct apply`, changes converge
4. **verify** - user can re-run plan to confirm no drift

## mental model

### how users would describe this to a friend

> "it's like terraform but for github teams. you declare what teams and members you want, and it makes github match that."

### analogies and metaphors

- **terraform for github teams** - declarative, plan/apply, state management
- **git for org structure** - version controlled, auditable, reviewable
- **recipe vs cook** - declare the outcome, tool figures out the steps

### terms: user language vs our language

| user says | we say | notes |
|-----------|--------|-------|
| "team" | `DeclaredGithubTeam` | the team itself |
| "member" | `DeclaredGithubTeamMembership` | user's membership in a team |
| "add someone" | `findsert` membership | idempotent add |
| "remove someone" | delete membership | remove from team |
| "make them maintainer" | `upsert` with role change | role promotion |
| "nested team" | team with `parent` ref | hierarchy |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| team existence | yes | create/update/delete teams |
| membership | yes | add/remove/change role |
| audit trail | yes | git history + declastruct state |
| nested teams | yes | parent reference |
| idempotency | yes | findsert/upsert semantics |

### pros

- **declarative** - state-as-code, not imperative scripts
- **auditable** - git blame shows who changed what
- **reviewable** - PRs for org changes
- **idempotent** - safe to re-apply
- **composable** - teams can reference other resources (org, parent team)

### cons

- **learn curve** - new abstraction to learn
- **state management** - need to understand declastruct's state model
- **permissions** - requires org admin or team maintainer permissions

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| user not in org | fail loud with clear error |
| circular parent reference | domain object validation rejects |
| secret team with parent | API constraint: parent teams can't be secret - validate at domain object level |
| delete team with children | API handles: children become orphans (top-level) |
| remove last maintainer | API constraint: teams need at least one maintainer - fail loud |

## groundwork

### external research

**GitHub Teams API** ([docs](https://docs.github.com/en/rest/teams/teams)):
- `POST /orgs/{org}/teams` - create team
- `GET /orgs/{org}/teams/{team_slug}` - get by slug
- `PATCH /orgs/{org}/teams/{team_slug}` - update
- `DELETE /orgs/{org}/teams/{team_slug}` - delete

**GitHub Team Members API** ([docs](https://docs.github.com/en/rest/teams/members)):
- `PUT /orgs/{org}/teams/{team_slug}/memberships/{username}` - add/update
- `DELETE /orgs/{org}/teams/{team_slug}/memberships/{username}` - remove
- roles: `member` or `maintainer`

**Key constraints verified**:
- privacy: `secret` or `closed` (not `public`)
- parent teams cannot be `secret`
- notification_setting: `notifications_enabled` or `notifications_disabled` (we simplify to `notifications: 'enabled' | 'disabled'`)
- membership roles: `member` or `maintainer`

## what is awkward?

### membership as separate resource

`DeclaredGithubTeamMembership` as a separate resource means:
- N resources for N members (verbose)
- harder to see "who's on this team" at a glance
- but: allows granular control, matches how other declastruct resources work

alternative: embed members in team:
```typescript
DeclaredGithubTeam.as({
  ...,
  members: [
    { username: 'alice', role: 'maintainer' },
    { username: 'bob', role: 'member' },
  ],
});
```
this is simpler but less composable and harder to manage at scale.

**recommendation**: start with separate `DeclaredGithubTeamMembership` resource - matches extant patterns and offers more flexibility.

### team slug uniqueness

team slugs are unique per org. if user renames team via UI, our config becomes stale.
- not a new problem - same for repos, branches, etc.
- solution: document that slug is the stable identifier

### parent team chicken-and-egg

if parent and child declared in same apply:
- parent must be created first
- declastruct should handle this via dependency order (refs)

**assumption**: declastruct handles ref-based order. need to verify.
