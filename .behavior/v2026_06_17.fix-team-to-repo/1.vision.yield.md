# vision: DeclaredGithubTeamRepoAccess

## the outcome world

### before/after contrast

**before:**
```ts
const team = DeclaredGithubTeam.as({
  org: { login: 'ehmpathy' },
  slug: 'releasers',
  // ...
});

const env = DeclaredGithubEnvironment.as({
  repo,
  name: 'production-on-else',
  reviewers: { users: null, teams: ['releasers'] },
  // ...
});

return [repo, team, env];
// ❌ declastruct apply fails with 422:
//    "Failed to create the environment protection rule.
//     Please ensure the billing plan supports the required reviewers protection rule."
//
// root cause: team has no repo access, so can't be reviewer
```

**after:**
```ts
const team = DeclaredGithubTeam.as({
  org: { login: 'ehmpathy' },
  slug: 'releasers',
  // ...
});

const teamRepoAccess = DeclaredGithubTeamRepoAccess.as({
  team: { org: { login: 'ehmpathy' }, slug: 'releasers' },
  repo,
  permission: 'push',
});

const env = DeclaredGithubEnvironment.as({
  repo,
  name: 'production-on-else',
  reviewers: { users: null, teams: ['releasers'] },
  // ...
});

return [repo, team, teamRepoAccess, env];
// ✅ works! team has repo access, then can be reviewer
```

### the "aha" moment

the moment you realize: "oh, teams need repo access before they can be environment reviewers, and now i can declare that relationship in the same place i declare everything else."

### day-in-the-life

1. devops engineer adds new team as environment reviewer
2. declastruct plan shows: "team 'releasers' will be added to repo with 'push' permission"
3. declastruct apply grants team access, then configures environment
4. done - team can now approve deployments

no manual github UI clicking. no terraform. just declare and apply.

## user experience

### usecases

| usecase | who | goal |
|---------|-----|------|
| team reviewers | devops | teams can approve environment deployments |
| team contributors | eng lead | teams can push/merge to repos |
| team readers | security | audit teams can read private repos |
| team admins | platform | platform team manages repo settings |

### contract inputs & outputs

**input contract:**
```ts
DeclaredGithubTeamRepoAccess.as({
  team: RefByUnique<typeof DeclaredGithubTeam>,  // { org, slug }
  repo: RefByUnique<typeof DeclaredGithubRepo>,  // { owner, name }
  permission: 'pull' | 'push' | 'triage' | 'maintain' | 'admin',
});
```

**unique key:** `[team, repo]` (a team can only have one permission level per repo)

**no primary key:** github doesn't return an id for this relationship - it's purely relational

### example declarations

```ts
// minimal: read-only access
const teamRepoAccess = DeclaredGithubTeamRepoAccess.as({
  team: { org: { login: 'ehmpathy' }, slug: 'auditors' },
  repo: { owner: 'ehmpathy', name: 'svc-payments' },
  permission: 'pull',
});

// common: push access for contributors
const teamRepoAccess = DeclaredGithubTeamRepoAccess.as({
  team: { org: { login: 'ehmpathy' }, slug: 'backend-engineers' },
  repo,
  permission: 'push',
});

// elevated: admin for platform team
const teamRepoAccess = DeclaredGithubTeamRepoAccess.as({
  team: { org: { login: 'ehmpathy' }, slug: 'platform' },
  repo,
  permission: 'admin',
});
```

### timeline

1. **declare** - add `DeclaredGithubTeamRepoAccess` to resources array
2. **plan** - declastruct shows what will change
3. **apply** - github API called, team granted access
4. **verify** - team members can now interact with repo per permission

## mental model

### how users describe it

> "i'm giving the releasers team push access to this repo"

> "teams need to be added to the repo before they can be reviewers on environments"

### analogies

- **like granting IAM roles**: you declare who (team) can do what (permission) to which (repo)
- **like adding collaborators**: but for teams, not individuals
- **like database grants**: `GRANT SELECT ON table TO role`

### terminology

| we call it | users might call it |
|------------|---------------------|
| `DeclaredGithubTeamRepoAccess` | "team repo permissions", "team access" |
| `permission: 'push'` | "write access", "contributor access" |
| `permission: 'pull'` | "read access", "read-only" |
| `permission: 'admin'` | "admin access", "full access" |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| team reviewers on environments | yes | prerequisite for `reviewers.teams` |
| declarative team access | yes | follows declastruct pattern |
| idempotent | yes | PUT is idempotent by nature |

### pros

- **unblocks team reviewers** - the actual problem that prompted this
- **follows extant patterns** - same shape as `DeclaredGithubTeamMembership`
- **declarative** - one source of truth for team repo access
- **idempotent** - safe to re-apply

### cons

- **no delete support yet** - github API supports removing access, but declastruct doesn't track deletions
- **requires org ownership** - repo must be in same org as team
- **permission escalation risk** - admins can grant themselves more access (inherent to github, not our problem)

### edge cases & pit of success

| edge case | handling |
|-----------|----------|
| team doesn't exist | declastruct apply fails (dependent ordering) |
| repo doesn't exist | declastruct apply fails (dependent ordering) |
| permission not in enum | typescript compile error |
| team in different org than repo | github API returns 422 - could add validation |
| custom repo roles | out of scope for v1 (string enum only) |

## open questions & assumptions

### assumptions

1. **teams are always org-scoped** - a team belongs to one org, and can only be added to repos in that same org
2. **no custom repo roles** - we only support standard permission levels, not org-defined custom roles
3. **upsert semantics** - PUT replaces prior permission (github PUT behavior)
4. **no order enforcement** - declastruct doesn't enforce order between resources; if team doesn't exist, github fails with clear error
5. **no org/owner validation** - let github enforce; pre-validation could have false positives (forks, personal repos)

### questions for wisher

1. **delete support?** - [wisher] should we track and remove access when removed from declarations?

### questions for research

none - github API is well documented.

## groundwork

### external research

**github API verified:**
- endpoint: `PUT /orgs/{org}/teams/{team_slug}/repos/{owner}/{repo}`
- permission levels: `pull`, `push`, `triage`, `maintain`, `admin`
- returns 204 on success (no body)
- source: https://docs.github.com/en/rest/teams/teams#add-or-update-team-repository-permissions

**constraints noted:**
- authenticated user must have admin access to repo
- repo must be owned by the org (or direct fork)
- returns 422 if repo not owned by org

### internal research

**extant patterns:**

1. **DeclaredGithubTeamMembership** (`src/domain.objects/DeclaredGithubTeamMembership.ts:10-37`)
   - same pattern: links team to something (user vs repo)
   - unique key: `[team, username]` → ours: `[team, repo]`
   - no primary key (github doesn't give id)

2. **DeclaredGithubTeam ref** (`src/domain.objects/DeclaredGithubTeamRefLiterals.ts`)
   - reuse `DeclaredGithubTeamRefLiteral` for nested team reference

3. **DeclaredGithubRepo ref** (`src/domain.objects/DeclaredGithubEnvironment.ts:20`)
   - uses `RefByUnique<typeof DeclaredGithubRepo>` with `{ owner, name }`

4. **acceptance tests** (`src/contract/sdks/.test/assets/resources.acceptance.ts`)
   - add new resource type to `getResources()` return array

## what is awkward?

### naming

`DeclaredGithubTeamRepoAccess` is accurate but verbose. alternatives considered:
- `DeclaredGithubTeamRepo` - too vague, sounds like "a team's repo"
- `DeclaredGithubTeamPermission` - doesn't say "on what"
- `DeclaredGithubRepoTeamAccess` - repo-first vs team-first

**decision:** keep `DeclaredGithubTeamRepoAccess` - it's clear and follows `{actor}{resource}{relationship}` pattern

### team ref in repo owner context

the repo's `owner` is a string, but the team's `org` is a ref. this means:
```ts
DeclaredGithubTeamRepoAccess.as({
  team: { org: { login: 'ehmpathy' }, slug: 'releasers' },  // org as ref
  repo: { owner: 'ehmpathy', name: 'my-repo' },              // owner as string
  permission: 'push',
});
```

the inconsistency is mildly awkward but matches extant contracts. no change needed.

### no way to verify team exists in same org

github will 422 if you try to add a team to a repo in a different org, but we could catch this earlier at declaration time. for v1, we let github enforce this - good enough.
