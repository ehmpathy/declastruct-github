# blocker.2: SDK positive path snapshots are missing

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.verification/rule.require.contract-snapshot-exhaustiveness.md

**locations**:
- src/contract/sdks/__snapshots__/declastruct.acceptance.test.ts.snap

The acceptance test snapshot file includes snapshots for SDK not-found paths (returning null) but no snapshots for positive paths where resources are found. For example, when calling getRepo with an existing repo, the response should be snapped. This violates the requirement for exhaustive snapshot coverage of all contract output variants (positive, negative, edge).

# blocker.3: CLI error messages expose raw stack traces and internal paths

**rule**: .agent/repo=ehmpathy/role=ergonomist/briefs/rule.forbid.surprises.md

**locations**:
- src/contract/sdks/__snapshots__/declastruct.acceptance.test.ts.snap

The error snapshot snapshots show raw stack traces with internal paths like '<repo>/node_modules/...' and function locations. These are not user-friendly error messages. Users encountering these errors will be surprised and confused. According to rule.forbid.surprises, error messages should be clear and not expose internal implementation details. The CLI should provide user-friendly error messages and log stack traces only in debug mode.

Examples:
- 'SyntaxError: Unexpected end of JSON input\n    at JSON.parse (<anonymous>)\n    at ...'
- 'Error: Transform failed with 1 error:\n<repo>/src/...'
These messages leak internal structure and are not actionable by typical users.
