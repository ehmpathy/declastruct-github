# vision: fix environment team reviewer provision

## the outcome world

### before/after contrast

**before:**
```
○ [CREATE] DeclaredGithubEnvironment.production-on-else.ownerehmpathy...
GET /orgs/ehmpathy/teams/releasers - 404 with id ...
✖ apply failed: BadRequestError: reviewer team 'releasers' not found in org 'ehmpathy'
```

developer stares at error, confused. the team exists. they created it. they can see it in GitHub UI. yet declastruct says "not found."

**after:**

provision just works. environments with team reviewers are created without 404 errors. the CI token has the permissions it needs.

### the "aha" moment

"oh, the CI token can't read org teams because it's repo-scoped. we need to use a token with org read permissions."

### day-in-the-life

1. developer declares an environment with team reviewers in their `provision/github.repo/resources.ts`
2. CI runs `declastruct apply`
3. environment is created with the team as a required reviewer
4. no permission errors, no confusion

## groundwork

### external research

**GitHub API docs for Get Team by Name:**
- endpoint: `GET /orgs/{org}/teams/{team_slug}`
- requires: organization membership for private teams, or `read:org` scope
- source: https://docs.github.com/en/rest/teams/teams#get-a-team-by-name

**GITHUB_TOKEN in GitHub Actions:**
- scoped to the repository, not the organization
- cannot access organization-level resources like teams
- source: https://docs.github.com/en/actions/security-for-github-actions/security-guides/automatic-token-authentication

**key insight:** `GITHUB_TOKEN` cannot read org teams. need either:
1. a GitHub App installation token with org read permissions
2. a PAT with `read:org` scope
3. a fine-grained PAT with organization read access

### internal research

**getTeamIdBySlug.ts:10-11:**
```typescript
 * .note = requires read:org scope on the GitHub token
```
the code already documents this requirement.

**setEnvironment.ts:43:**

this is where the lookup happens at environment provision time.

**resources.acceptance.ts:99-102:**
```typescript

  teams: null,  // <-- acceptance tests avoid team reviewers!

```
the acceptance tests sidestep the issue by not referencing team reviewers.

**howto.test-admin-resources.md (brief):**
documents that team-related resources require `admin:org` scope which CI/CD tokens lack.

## what is awkward?

### the error message is unclear

```
BadRequestError: reviewer team 'releasers' not found in org 'ehmpathy'
```

this says "not found" but the team exists. the real issue is "no permission to read teams." users waste time to verify the team exists when the problem is permissions.

### the fix path is unclear

when a user hits this error, what should they do?
- use a different token?
- add a GitHub App?
- give up on team reviewers?

the error doesn't guide them.

### acceptance tests don't cover team reviewers

the acceptance tests explicitly avoid `teams: ['...']` reviewers, so this code path has no automated coverage. the issue only surfaces when users try to use it in production.

## proposed direction

### root cause

the CI token (`GITHUB_TOKEN`) cannot read org teams. `getTeamIdBySlug` returns 404, which is interpreted as "team not found" rather than "no permission."

### fix options

1. **improve error message** — detect 404 on team lookup and suggest the permission requirement
2. **document the requirement** — make it clear that team reviewers need elevated token permissions
3. **support team IDs directly** — allow `teamIds: number[]` to bypass slug lookup (escape hatch)
4. **recommend GitHub App usage** — guide users to use apps with org read permissions

### recommended approach

1. **short term:** improve error message to explain permission requirement
2. **medium term:** document the token requirements clearly
3. **long term:** consider if there's a way to provision team reviewers without org read requirement (unlikely given GitHub API design)
