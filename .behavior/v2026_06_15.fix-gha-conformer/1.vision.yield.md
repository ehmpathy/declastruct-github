# vision: fix-gha-conformer

## the outcome world

**before:** the `declastruct-github-conformer` app can't read environment data. when declastruct plans changes involving `DeclaredGithubEnvironment` resources, the plan check fails with a permissions error. automerge is blocked.

**after:** the app has `environments: read` permission. plan checks pass. automerge proceeds normally.

**aha moment:** PRs that touch environment resources no longer stall on the plan check.

## user experience

| usecase | what happens |
|---------|--------------|
| plan environments | `getEnvironment` succeeds, plan shows correct diff |
| apply environments | `setEnvironment` can read before state, upsert works |
| automerge | plan check passes, PR merges automatically |

**contract change:**

```diff
// provision/github.apps/resources.app.declastruct-github.conformer.ts
permissions: {
  repository: {
    administration: 'write',
    metadata: 'read',
+   environments: 'read',
  },
  organization: {
    members: 'read',
  },
},
```

**timeline:**
1. add permission to app declaration
2. run `declastruct apply` on `provision/github.apps/resources.ts`
3. github app gains permission
4. plan checks pass

## mental model

> "the conformer app needs to read environments to manage them — same pattern as needing `metadata: read` to see repos."

analogy: you can't rearrange furniture in a room you can't enter. `environments: read` lets the app see what's there before deciding what to change.

## evaluation

**pros:**
- minimal permission scope (`read` not `write`)
- unblocks automerge
- consistent with extant patterns (`metadata: read`, `members: read`)

**cons:**
- none significant — this is narrowly scoped

**edgecases:**
- `setEnvironment` already has write capability via `administration: write`
- `read` is sufficient; we only need it for `getEnvironment` lookups

## open questions & assumptions

**assumptions:**
- `environments: read` is sufficient (not `write`)
- `administration: write` covers create/update/delete of environments
- the failing check is specifically due to missing read permission

**questions:**
- none — the error message was explicit

**validation needed:**
- none — the fix is mechanical

## groundwork

### external research

GitHub API docs confirm:
- `environments: read` grants `GET /repos/{owner}/{repo}/environments/{name}`
- `environments: write` (or `administration: write`) grants create/update/delete
- ref: https://docs.github.com/en/rest/deployments/environments

### internal research

verified in codebase:
- `getEnvironment.ts` calls `github.repos.getEnvironment` — needs read permission
- `setEnvironment.ts` calls `getEnvironment` before upsert — needs read permission
- `DeclaredGithubAppRepositoryPermissions` already defines `environments?: GithubAppPermissionLevel | null` (line 74)
- conformer app at `provision/github.apps/resources.app.declastruct-github.conformer.ts` currently lacks `environments` in repository permissions

## what is awkward?

none — this is a simple permission addition following established patterns. the only awkwardness is that we didn't include it when we added `DeclaredGithubEnvironment` support, but that's water under the bridge.
