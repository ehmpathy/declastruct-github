wish =

gotta allow tag deployments


  ---
  Handoff → declastruct-github: support type: 'tag' deployment policies

  Problem

  DeclaredGithubEnvironment cannot express a tag deployment policy. GitHub environments gate deploys by ref, and each custom policy carries a type: 'branch' | 'tag'. A tag-triggered deploy
  (e.g. refs/tags/v0.32.1) only matches type: 'tag' policies — never a branch policy, even one literally named v*.

  Today the domain type offers only:
  deploymentBranchPolicy: null
    | { protectedBranches: true }
    | { customBranches: string[] }
  and syncDeploymentBranchPolicies.js hardcodes the type:
  await github.repos.createDeploymentBranchPolicy({
    owner, repo, environment_name,
    name: pattern,
    type: 'branch',   // ← always branch; tags are unreachable
  });

  Real-world impact (ahbode/svc-notifications)

  Our prod deploy fires on v* tags cut from main. We set production-on-main with deploymentBranchPolicy: { customBranches: ['main'] } intending "deploys come from main." But release tags
  don't match a branch policy, so every prod deploy is rejected:
  Tag "v0.32.1" is not allowed to deploy to production-on-main due to environment protection rules.
  There is currently no way to declare "allow v* tags to deploy here" through declastruct-github. Our only workaround is deploymentBranchPolicy: null (allow any ref) — which drops the ref
  restriction from the env entirely, relying solely on the tag-trigger + a protect-release-tags ruleset. That works, but it's strictly less expressive than GitHub allows, and the env's own
  policy no longer documents intent.

  Requested change

  Let deploymentBranchPolicy carry per-pattern type, so both branch and tag custom policies can be declared. GitHub allows both custom_branch_policies and tags simultaneously (each policy row
   has its own type).

  Proposed domain shape (backwards-compatible — keep customBranches working):
  deploymentBranchPolicy: null
    | { protectedBranches: true }
    | { customBranches: string[] }                     // existing; implies type:'branch'
    | { customPatterns: Array<{ name: string; type: 'branch' | 'tag' }> }  // new
  Then asDeploymentBranchPolicyConfig should set custom_branch_policies: true for the new variant too, and syncDeploymentBranchPolicies should:
  - read extant policies via listDeploymentBranchPolicies (each row already reports type)
  - compare on the (name, type) pair, not name alone (so a v* branch vs v* tag are distinct)
  - create with the declared type instead of the hardcoded 'branch'
  - delete stale ones by id as it does today

  Acceptance criteria

  - an env declared with { customPatterns: [{ name: 'v*', type: 'tag' }] } provisions a tag policy; a v* tag deploy is allowed, a v* branch is not
  - mixed [{ name: 'main', type: 'branch' }, { name: 'v*', type: 'tag' }] provisions both rows
  - existing customBranches: [...] callers are unchanged (still create branch policies)
  - sync is idempotent: re-apply with the same desired set is a no-op; changing a pattern's type replaces the row (delete + create), not leaves a duplicate

  References (declastruct-github source)

  - dist/domain.objects/DeclaredGithubEnvironment.d.ts — the deploymentBranchPolicy union to extend
  - dist/domain.operations/environment/setEnvironment.js → asDeploymentBranchPolicyConfig (maps to custom_branch_policies)
  - dist/domain.operations/environment/syncDeploymentBranchPolicies.js:71-77 — the hardcoded type: 'branch' create call (and the name-only comparison at extractPatternNamesAsSet)
  - GitHub API: POST /repos/{owner}/{repo}/environments/{env}/deployment-branch-policies accepts { name, type: 'branch' | 'tag' }; the env must have
  deployment_branch_policy.custom_branch_policies: true

  Note on the ruleset (already correct)

  protect-release-tags (in our repo, DeclaredGithubRepoRuleset, target: 'tag') already handles who may cut v* tags. This request is orthogonal — it's about which refs an environment accepts
  for deployment. Both are needed for the full gate; only the environment side is missing tag support.
