# 1.vision — support `target: 'tag'` deployment policies

> wish: `.behavior/v2026_07_20.fix-env-deploy-policy/0.wish.md`
> "gotta allow tag deployments"

---

## the outcome world

### day-in-the-life

a team ships prod on release tags. their pipeline cuts `v0.32.1` from `main`, then
triggers a deploy to the `production-on-main` environment.

**before** — the env's ref gate only understands branches. the tag matches none:

```
Tag "v0.32.1" is not allowed to deploy to production-on-main
due to environment protection rules.
```

their only escape is `deploymentBranchPolicy: null` (allow *any* ref) — which throws
away the ref restriction entirely and leaves the env's own policy silent about intent.

**after** — they declare exactly what they mean:

```ts
DeclaredGithubEnvironment.as({
  repo,
  name: 'production-on-main',
  reviewers: null,
  waitTimer: null,
  deploymentBranchPolicy: {
    customPatterns: [{ name: 'v*', target: 'tag' }],
  },
  preventSelfReview: false,
});
```

`declastruct apply` provisions a **tag** policy row. the `v0.32.1` deploy is allowed.
a branch literally named `v-something` is *not* — because the policy is scoped to tags.

### before / after contrast

| dimension | before | after |
|-----------|--------|-------|
| tag deploys | impossible to declare | first-class via `target: 'tag'` |
| workaround | `deploymentBranchPolicy: null` (all refs) | precise per-pattern gate |
| intent in code | lost (env allows everything) | documented (`v*` tags only) |
| branch + tag mix | not expressible | `customPatterns` carries both |
| branch-only envs | `customBranches: [...]` | restated as `customPatterns: [{ target: 'branch' }]` (hardcut) |

### the "aha" moment

the value clicks when a user reads the env declaration months later and *sees the gate*:
"deploys here come from `v*` tags." the declaration is the documentation. no tribal
knowledge about tag-triggers + rulesets papering over a `null` policy.

## user experience

### usecases / goals

1. **tag-only prod deploys** — allow `v*` tags, reject everything else. (the ahbode/svc-notifications case.)
2. **mixed branch + tag** — allow `main` branch deploys *and* `v*` tag deploys to the same env.
3. **branch-only envs** — declare `customPatterns: [{ name: 'main', target: 'branch' }]` (the hardcut replacement for `customBranches`).

### contract inputs

**wisher decision (2026-07-20): hardcut.** `customBranches` is dropped entirely and replaced by
`customPatterns` — no dual-shape, no sugar, no backwards-compat variant. the feature is young
("green"), so a clean cut beats a carried synonym. this collapses the whole round-trip / option
A-vs-B fork into one canonical design.

the `deploymentBranchPolicy` union becomes:

```ts
deploymentBranchPolicy:
  | null                                                     // all refs
  | { protectedBranches: true }                              // protected branches only
  | { customPatterns: Array<{ name: string; target: 'branch' | 'tag' }> };  // branch and/or tag rows
```

> **vocab note (`target`, not `type`)**: the wish proposed `type: 'branch' | 'tag'` to mirror
> GitHub's API field. but the codebase already calls this exact branch-vs-tag axis `target` —
> `DeclaredGithubRepoRuleset.target: 'branch' | 'tag'` (`DeclaredGithubRepoRuleset.ts:74`).
> a reused `type` would be synonym drift (`rule.require.ubiqlang` / no-synonyms), and `type` is
> an overloaded term besides — the wisher's "rather cut than have synonyms while it's so green"
> settles it: the domain field is **`target`**; the cast maps it to GitHub's wire `type` at the
> API boundary.

- `customPatterns: [{ name: 'main', target: 'branch' }]` — a branch-only gate (the old `customBranches: ['main']`).
- `customPatterns: [{ name: 'v*', target: 'tag' }]` — a tag gate.
- `customPatterns: [{ name: 'main', target: 'branch' }, { name: 'v*', target: 'tag' }]` — both rows.

> **migration**: every extant `customBranches: [...]` caller restates as
> `customPatterns: [{ name, target: 'branch' }, ...]`. in-repo that is the acceptance resources
> (`resources.acceptance.ts`); downstream repos (e.g. ahbode/svc-notifications) migrate on upgrade.
> a contract break, deliberately taken while the surface is small.

### contract outputs

`getEnvironment` returns a `DeclaredGithubEnvironment` whose `deploymentBranchPolicy`
reflects the live GitHub state, cast back through `castToDeclaredGithubEnvironment`. with the
hardcut, the cast always returns `customPatterns` (never `customBranches`), so declared and got
share one shape — the round-trip converges by construction, no conditional-cast rule needed.
each GitHub policy row already reports its own `type`, which the cast reads into `target`.

### what leveraging it looks like

```ts
// declare
const env = DeclaredGithubEnvironment.as({
  repo,
  name: 'production-on-main',
  reviewers: null,
  waitTimer: null,
  deploymentBranchPolicy: { customPatterns: [{ name: 'v*', target: 'tag' }] },
  preventSelfReview: false,
});

// plan + apply (via declastruct)
//   → POST deployment-branch-policies { name: 'v*', type: 'tag' }  (wire field is `type`)
//   → env.deployment_branch_policy.custom_branch_policies = true

// re-apply → no-op (idempotent)
```

### timeline

1. declare env with `customPatterns`.
2. `declastruct plan` shows a create for the `(v*, tag)` policy.
3. `declastruct apply` provisions it (env flag `custom_branch_policies: true` + policy row).
4. a `v*` tag deploy is accepted; a `v*` branch is rejected.
5. re-apply is a no-op; a change to a pattern's `target` replaces the row (delete + create).

## mental model

### how a user describes it to a friend

> "GitHub environments can gate deploys by branch *or* by tag. declastruct used to only
> let me declare branch gates, so my tag-triggered prod deploys got rejected. now I just
> say `{ name: 'v*', target: 'tag' }` and it works."

### analogies / metaphors

- a **bouncer's guest list** with two columns: "branches" and "tags." before, we could
  only write names in the branches column, so tag-holders were turned away even if their
  name matched a branch entry. now each entry says which column it belongs to.
- like a **customs form** where each item declares its category — a `v*` in the "tag"
  lane clears; the same string in the "branch" lane is a different declaration.

### terms: theirs vs ours

| they say | we say |
|----------|--------|
| "allow tag deploys" | `customPatterns: [{ target: 'tag' }]` |
| "deploy branch policy" | `deploymentBranchPolicy` |
| "release tag" | a `tag` pattern (e.g. `v*`) |
| "the env rejected my deploy" | ref did not match a policy of that ref's target |

`customPatterns` is our term; GitHub calls each row a "deployment branch policy" (even for
tags, confusingly). we keep `deploymentBranchPolicy` as the domain field to match extant
naming; the sole custom variant carries a per-row `target`, which the cast maps to the
per-row `type` GitHub already models on the wire.

## evaluation

### how well it solves the goals

- **tag deploys**: solved at the layer we own — declastruct *provisions* a `target: 'tag'`
  policy row. ✓ (see the verifiability caveat below: the deploy-gate *behavior* is GitHub's.)
- **mixed branch+tag**: solved — array of typed patterns. ✓
- **one canonical shape**: the hardcut leaves a single custom variant, so declared and got
  always match — the round-trip converges by construction. ✓
- **idempotency**: solved by comparing on the `(name, target)` pair, not name alone. ✓

### verifiability caveat — what "acceptance" can prove here

the wish's acceptance criterion reads "a `v*` tag deploy is **allowed**, a `v*` branch is
**not**." declastruct only *provisions* the policy; it never *triggers a deploy*. so the
provision-side acceptance suite can verify **the tag policy row exists with `target: 'tag'`**,
but it cannot verify GitHub's deploy gate actually admits the tag / rejects the branch — that
is GitHub's runtime behavior, out of declastruct's control. the honest split: our acceptance
proves the row is provisioned correctly; the deploy-gate behavior is validated separately
(the real ahbode deploy, or a manual check). the vision's acceptance ambition must stop at
"provisions the right row," not "GitHub admits the ref."

### pros

- strictly more expressive; matches what GitHub's API already allows.
- **one canonical shape** — the hardcut removes the dual-representation entirely; a reader learns
  exactly one way to declare a custom gate.
- intent lives in the declaration (self-documenting gate).
- reuses the extant sync/create/delete machinery; the `target` value and the comparison key change.

### cons / tradeoffs

- **a contract break** — every `customBranches` caller must migrate. accepted by the wisher
  because the feature is young and the caller set is small.
- GitHub's naming ("deployment *branch* policy" for a tag row) leaks confusion we inherit.

### edgecases & pit-of-success

| edgecase | how the contract keeps users safe |
|----------|-----------------------------------|
| `v*` tag vs `v*` branch collision | distinct rows — compare on `(name, target)`, so both can coexist and neither shadows the other |
| change a pattern's `target` | not an in-place update on GitHub — must delete old row + create new; sync does this by a diff of the `(name, target)` pair |
| declare `customPatterns` but env flag off | `asDeploymentBranchPolicyConfig` must set `custom_branch_policies: true` for the new variant too, else the POST 422s |
| empty `customPatterns: []` | reject at construction (fail-fast) — an ambiguous no-op; steer the caller to `null` or a non-empty list |
| duplicate `(name, target)` in input | domain constructor should reject to prevent a confusing double-create — **proposed guard** |
| partial sync failure on **update** | `setEnvironment.ts:160-182` only rolls back when it *created* the env (`wasCreated`); an update whose sync 422s mid-way leaves some rows created, some not, no rollback. two targets widen this surface — the `(name, target)` create loop could fail after N rows. sync should stay order-independent + idempotent so a re-apply heals a partial failure — **must confirm re-apply converges from a partial state** |

## open questions & assumptions

### the central design question: **resolved by the hardcut**

earlier drafts wrestled with what `getEnvironment` casts back to, because two custom shapes
(`customBranches` vs `customPatterns`) could diverge and cause a **perpetual diff** — the plan
would never converge. the wisher's hardcut (2026-07-20) removes the dilemma: `customBranches` is
gone, so there is exactly one custom shape. `castToDeclaredGithubEnvironment` always returns
`customPatterns` (each row's wire `type` read into `target`), the declared value is always
`customPatterns`, and the two match by construction. no conditional-cast rule, no `asCustomPatterns`
sugar transformer, no snapshot-tolerance question. the one price — the extant
`castToDeclaredGithubEnvironment.ts` path that returns `{ customBranches }` (`:113-131`) is
rewritten to emit `customPatterns` — is exactly the contract break the wisher signed off on.

> **outcome:** the hardcut *is* the old "option B" taken to its end — one canonical
> `customPatterns` shape — minus the sugar layer that would have kept `customBranches` alive.
> the wisher chose the cleaner cut ("rather cut than have synonyms while it's so green"), so
> there is no remaining design fork here.

### triaged open questions

each question is tagged `[answered]` (settled now via logic/code), `[research]` (needs an
external check at execution), or `[wisher]` (only the wisher can decide).

1. **`[answered]` behavior for `customPatterns: []`** — reject at construction (fail-fast).
   an empty custom-pattern set is an ambiguous no-op: it neither declares a gate nor clears
   one. per `rule.require.failfast`, the constructor should `BadRequestError.throw` and steer
   the caller to `null` (all refs) or a non-empty list. no wisher input needed.

2. **`[answered]` acceptance envs after the hardcut** — every extant acceptance env migrates
   from `customBranches: ['main']` to `customPatterns: [{ name: 'main', target: 'branch' }]`
   (forced by the contract break). additionally add a *new* tag env
   (`customPatterns: [{ name: 'v*', target: 'tag' }]`) and a mixed branch+tag env, so the suite
   exercises both targets. a logic-level test-design call, not a wisher decision.

3. **`[answered]` domain field name: `target`, not `type`** — settled this pass. the codebase
   already names the branch-vs-tag axis `target` (`DeclaredGithubRepoRuleset.ts:74`); the
   domain uses `target`, the cast maps to GitHub's wire `type`. see the vocab note above.

4. **`[answered by wisher]` drop `customBranches` entirely (hardcut)?** — **yes** (2026-07-20).
   the wisher chose a clean contract break over a dual-shape / sugar layer while the feature is
   young. this settles the round-trip design: one canonical `customPatterns` shape, no option
   fork, snapshot churn accepted.

5. **`[answered]` [was: critical premise] can an env hold branch AND tag rows at once under one
   `custom_branch_policies: true` flag?** **yes — confirmed against GitHub docs (2026-07-20).**
   the feature shipped 2023-10-06 ("secure deployment rollouts ... based on select tag
   patterns"); the UI exposes a per-row "Ref type" dropdown of Branch/Tag, patterns are set for
   branches or tags individually, and the docs describe a restriction over "which branches
   **and** tags can deploy" to one env gated by the single `custom_branch_policies: true` flag.
   the mixed-env requirement is real, not hopeful.
   - sources: docs.github.com/en/rest/deployments/branch-policies ；
     github.blog/changelog/2023-10-06-actions-secure-deployment-rollouts-to-protected-environments-based-on-select-tag-patterns

6. **`[research]` does `listDeploymentBranchPolicies` return each row's `type`?** drives whether
   the sync compare + cast-back need an `as any` (like the extant protection-rule casts) or an
   upstream type contribution.

7. **`[research]` does octokit's create param type-check with `type: 'tag'`?** the param exists
   (extant code passes `type: 'branch'`); only the *value* is new. cheap to confirm; fallback
   is a documented cast.

8. **`[research]` how does declastruct's plan compare `deploymentBranchPolicy`?** full
   field-by-field on the nested value, or unique-key only? less critical after the hardcut (one
   canonical shape means declared and got match regardless), but still worth a confirm so a
   nested-value change (e.g. a pattern added) is actually detected as drift and not silently
   skipped.

9. **`[answered]` do `reviewers` / `waitTimer` protections apply to tag deploys?** yes — these
   are separate protection rules on the env, orthogonal to the ref (`deploymentBranchPolicy`)
   gate. a tag that clears the ref gate still faces required reviewers + wait timer exactly as a
   branch does. this means the `target: 'tag'` change is self-contained: it touches only the ref
   dimension and needs no change to `lookupReviewerIds` or the wait-timer path in
   `setEnvironment.ts`. answerable by logic from the extant protection-rule model; no wisher or
   research needed.

  deployment branch **or tag** policy" and requires `deployment_branch_policy.custom_branch_policies: true`.
  the tag capability shipped 2023-10-06; the UI exposes a per-row "Ref type" (Branch/Tag), and one
  env may restrict "which branches **and** tags can deploy" at once — so branch+tag coexistence
  under the single flag is confirmed, not assumed.

  **public** repos; private repos need GitHub Pro/Team. the acceptance demo

## what is awkward?

1. **the contract break has downstream cost.** the hardcut is clean in-repo but every
   downstream `customBranches` caller (e.g. ahbode/svc-notifications) breaks on upgrade until
   migrated. the wisher accepted this because the surface is young and small — the discomfort is
   real but bounded, and the alternative (a permanent synonym) is worse.
2. **the round-trip was the hard part — now dissolved.** the earlier draft's cast-back
   idempotency risk (a declared `customPatterns` vs a got `customBranches` never converging) is
   gone: one canonical shape means declared and got match by construction. what remains is
   mechanical (rewrite the cast to emit `customPatterns`), not a design puzzle.
3. **GitHub's naming leaks**: "deployment *branch* policy" that holds *tag* rows. we inherit
   the confusion; `customPatterns` + `target` soften it, but the field stays
   `deploymentBranchPolicy`.
4. **`target` change = delete + create**, not update. GitHub has no in-place row-type edit, so
   the sync's idempotency story has a "replace" path that's slightly more than a pure diff.
   surfaced as an edgecase; feels forced but is inherent to the API.
