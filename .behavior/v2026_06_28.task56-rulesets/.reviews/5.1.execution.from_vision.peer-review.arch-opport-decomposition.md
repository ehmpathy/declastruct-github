# blocker.1: Raw SDK calls in orchestrators instead of communicators

**rule**: rule.forbid.decode-friction-in-orchestrators (from .agent/repo=ehmpathy/role=architect/briefs/practices/rule.forbid.decode-friction-in-orchestrators.md)

**locations**:

- src/domain.operations/repoRuleset/setRepoRuleset.ts:72-78
- src/domain.operations/repoRuleset/delRepoRuleset.ts:41-44

The orchestrators `getRepoRuleset`, `setRepoRuleset`, and `delRepoRuleset` directly invoke Octokit SDK methods (`github.repos.getRepoRulesets`, `github.repos.getRepoRuleset`, `github.repos.createRepoRuleset`, `github.repos.updateRepoRuleset`, `github.repos.deleteRepoRuleset`) without wrapping them in domain-named communicators.

The rule states that raw API calls (e.g., `await stripe.customers.create(...)`) must be extracted to a communicator (e.g., `sdkStripe.setCustomer(...)`) to avoid decode‑friction and to maintain a clean orchestrator narrative. Each orchestrator should read as a sequence of *what* happens, not *how* the SDK is called.

For each raw call, introduce a communicator such as `getGithubRepoRuleset`, `setGithubRepoRuleset`, `delGithubRepoRuleset` that encapsulates the Octokit invocation, error handling, and idempotency logic.
