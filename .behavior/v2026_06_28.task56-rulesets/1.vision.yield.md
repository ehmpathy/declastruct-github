# 1.vision — github rulesets as code (repo + org)

## the wish

fulfill radio task 56: add support for **github rulesets** at both **repo level** and **org level**, so consumers declare tag-protection (and branch) rulesets as code via the declastruct get/set pattern — mirroring extant `DeclaredGithubBranchProtection` + `DeclaredGithubEnvironment`.

---

## the outcome world

### day-in-the-life

vlad is wiring up the aws-oidc "prod apply only from a version tag cut from main" guarantee in `ahbode/infrastructure`. the aws half is done: sts conditions on `ref_type == tag` + `ref LIKE refs/tags/v*`. the github half is missing — he needs a tag ruleset that restricts who can create `v*` tags to the release-please github app only.

today he'd click through github org/repo settings → rules → rulesets, hand-craft a bypass actor, and pray nobody edits it out of band. there's no record in code, no diff, no review.

after this wish, he opens his `resources.ts`, adds a `DeclaredGithubRepoRuleset` next to the repo it guards, declares `target: 'tag'`, `conditions.refName.include: ['refs/tags/v*']`, a `creation` rule, and a bypass actor for the release app. he runs `declastruct plan`, sees the ruleset appear as a `CREATE` change, runs `declastruct apply`, and the guarantee is now codified, diffable, and review-gated.

### before / after

| | before | after |
|---|---|---|
| declare a tag ruleset | click-ops in github ui | `DeclaredGithubRepoRuleset.as({ ... })` in code |
| review a change | none — invisible | shows in `declastruct plan` diff + PR |
| drift detection | manual audit | `plan` reports drift idempotently |
| org-wide policy | repeat per repo by hand | one `DeclaredGithubOrgRuleset` with repo conditions |
| the oidc guarantee | half-codified (aws only) | fully codified (aws + github) |

### the "aha" moment

the value clicks when `declastruct plan` prints a `CREATE DeclaredGithubRepoRuleset` change for the `v*` tag protection — the same shape as the repos, environments, and branch-protections already in the file. rulesets stop being a special snowflake in the github ui and become just another declared resource in the same pit of success.

## user experience

### usecases & goals

1. **tag protection (primary)** — restrict `v*` tag creation to a release app (release-please) via a bypass actor. completes the aws-oidc prod-apply guarantee.
2. **branch rulesets** — declare modern branch protections (the ruleset successor to legacy branch-protection) at repo scope.
3. **org-wide rulesets** — apply a ruleset across many repos at once via `conditions.repository_name` include/exclude, instead of repeating per repo.

### contract inputs & outputs

consumers leverage two new domain objects, registered in the provider's `daos` and exported from `src/contract/sdks/index.ts`:

```ts
import { DeclaredGithubRepoRuleset, DeclaredGithubOrgRuleset } from 'declastruct-github';

// repo-scoped: restrict v* tag creation to the release app
const releaseTagRuleset = DeclaredGithubRepoRuleset.as({
  repo,                                  // RefByUnique<typeof DeclaredGithubRepo>
  name: 'protect-version-tags',          // natural unique key (per repo)
  target: 'tag',
  enforcement: 'active',
  conditions: { refName: { include: ['refs/tags/v*'], exclude: [] } },
  rules: [{ type: 'creation' }],
  bypassActors: [
    { actorId: 123456, actorType: 'Integration', bypassMode: 'always' }, // release-please app
  ],
});

// org-scoped: same intent, applied across selected repos
const orgTagRuleset = DeclaredGithubOrgRuleset.as({
  org,                                   // RefByUnique<typeof DeclaredGithubOrg>
  name: 'org-protect-version-tags',
  target: 'tag',
  enforcement: 'active',
  conditions: {
    refName: { include: ['refs/tags/v*'], exclude: [] },
    repositoryName: { include: ['~ALL'], exclude: [] },  // org-only condition
  },
  rules: [{ type: 'creation' }],
  bypassActors: [{ actorId: 123456, actorType: 'Integration', bypassMode: 'always' }],
});
```

outputs mirror extant resources: `get` returns `HasMetadata<DeclaredGithub*Ruleset> | null` (server-assigned `id` populated), `set.findsert`/`set.upsert` return the persisted resource after read-after-write.

### timeline of use

1. add `DeclaredGithubRepoRuleset` / `DeclaredGithubOrgRuleset` to `resources.ts` (and `resources.acceptance.ts` for our own coverage)
2. `declastruct plan` → review the `CREATE`/`UPDATE`/`no-op` diff
3. `declastruct apply` → ruleset provisioned via octokit
4. re-run `plan` → idempotent no-op proves convergence

## mental model

> "it's terraform for github rulesets — i declare the ruleset i want, it figures out create vs update by name, and `plan` shows me the diff before anything changes."

| analogy | fit |
|---|---|
| terraform resource | declare desired state, idempotent converge |
| the extant `DeclaredGithubBranchProtection` | same shape, rulesets are the modern successor |
| a guard at the gate | bypass actors = the only ones allowed past the rule |

**terms users use vs terms we use:**

| user says | we model as |
|---|---|
| "the v* tag rule" | `DeclaredGithubRepoRuleset` with a `creation` rule + `refName` condition |
| "let release-please through" | `bypassActors: [{ actorType: 'Integration', ... }]` |
| "ruleset id" | `id` (`@metadata`, server-assigned primary key) |
| "ruleset name" | `name` (natural unique key, per scope) |

## evaluation

### how well it solves the goals

- ✅ fully codifies the github half of the oidc guarantee (the originating need)
- ✅ symmetric repo + org scopes
- ✅ slots into the extant declastruct lifecycle with zero new consumer concepts

### pros

- mirrors a proven, house-blessed pattern (low novelty risk)
- octokit already ships every needed method (`repos.{get,create,update,delete}{Repo,Org}Ruleset` + `get{Repo,Org}Rulesets`) — verified installed
- `name`-keyed uniqueness gives a clean idempotency story (**assumes** github treats ruleset name as unique-per-scope — unverified, see assumptions below)

### cons / tensions

- **name → id resolution**: the github get-by-id endpoint needs the server-assigned `id`, which the unique key (`name`) doesn't carry. so `get.byUnique` must **list rulesets and filter by name** — a pattern not yet used in this repo (environments/branch-protection address resources by name directly in the URL). this is the one genuinely new mechanism.
- **rule/condition shape breadth**: rulesets support many rule types and parameters. v1 should cover the task's needs (`creation`, `update`, `deletion`, `required_signatures`, `required_linear_history`, `non_fast_forward`, plus `refName` conditions and bypass actors) and model the `rules[]`/`conditions`/`bypassActors[]` as nested literals — not chase 100% of github's surface.
- **bypass actor ids are opaque ints**: unlike environment reviewers (resolved from usernames/slugs), the task's api reference passes `actor_id` directly. v1 takes ids as-is (matches the api); name→id resolution for actors is out of scope unless the wisher wants it.

### edgecases & pit-of-success

| edgecase | how the contract keeps users safe |
|---|---|
| ruleset absent yet | `get.byUnique` returns `null` → `set` creates (findsert/upsert symmetry) |
| duplicate name in scope | list+filter expects exactly one — fail-fast (`UnexpectedCodePathError`) if >1. **note**: whether github *enforces* name uniqueness per scope is unverified (assumption A below); the fail-fast is the safety net if it does not |
| `get.byPrimary` (by id) | `undefined` like branch-protection, OR supported since the api has it — **open question below** |
| org-only conditions on a repo ruleset | type system forbids `repositoryName` on `DeclaredGithubRepoRuleset` (separate domain objects, not a shared one with optional fields) |
| 404 on get/delete | return `null` / no-op — idempotent, matches extant pattern |

## open questions & assumptions

### wisher decisions (2026-06-28)

the wisher answered the three **[wisher]** items:

- **delete support (Q1)** → **yes.** rulesets expose `set.delete` (like environments); the dao wires `delete` and a `delRuleset` operation.
- **name immutability / rename→duplicate risk (assumption B)** → **accept it**, same as `DeclaredGithubEnvironment`. the key stays `unique = ['repo','name']`; we do not key on `id`.
- **org-level sequence (Q5)** → **repo-only for v1.** `DeclaredGithubOrgRuleset` + its dao are deferred to a fast-follow. this delivery ships **`DeclaredGithubRepoRuleset` + dao + tests only**. the org `admin:org` CI constraint is therefore moot for v1, and all v1 tests run in CI.

these convert deliverable 2 (org ruleset) to a future item; the vision text below predates the decision and is kept for context.

### assumptions made

1. **mirror branch-protection/environment exactly** — file layout, cast functions, `ContextGithubApi & ContextLogTrail`, `asProcedure`, one-export-per-file, fail-fast, read-after-write. (the task explicitly asks for this.)
2. **two separate domain objects**, not one with an optional org-condition — keeps repo vs org type-safe and avoids adhoc optionality. org adds `conditions.repositoryName`.
3. **`primary = ['id']`, `unique = ['repo','name']`** (repo) and **`unique = ['org','name']`** (org) — per the task's key choices.
4. **bypass actor ids passed as-is** (no username/slug → id resolution in v1).
5. **`rules`, `conditions`, `bypassActors` modeled as nested `DomainLiteral`** inline-typed shapes (mirrors how branch-protection models `requiredStatusChecks` etc.), not separate domain objects — the task lists supporter value objects as optional ("if the repo favors that granularity"); the repo currently favors inline nested literals.

### hidden assumptions to verify (surfaced in self-review)

these were taken as fact in earlier drafts; they are actually unverified inferences:

- **(A) github treats ruleset `name` as unique-per-scope.** — **[research]** the entire unique-key design (`unique = ['repo','name']`) rests on this, yet i have **not** verified github *enforces* it at the api level — github ruleset names may be non-unique. answer in the research phase (attempt to create two same-named rulesets, or check docs). if not enforced: the fail-fast on >1 match is the safety net, but the key choice itself would need a rethink. *the task asserts "name = natural unique key per scope" as a design choice, but does not cite api enforcement.*
- **(B) ruleset `name` is immutable.** — **[wisher]** the house rule `rule.require.immutable-refs` requires unique-key fields to be immutable, but github ruleset names are **user-editable in the UI**. if a consumer renames a ruleset out-of-band, declastruct sees the old name as absent → creates a **duplicate**. this is the same property the extant `DeclaredGithubEnvironment` has (env names are also editable), so it is consistent with precedent — but it is an assumption, not a guarantee. wisher decides: accept the rename→duplicate risk (as environments do), or key on `id`?
- **(C) which integer is `actorId`.** — **[research]** the vision shows `actorId: 123456` for a github app and infers the app's **id**. github's bypass-actor `actor_id` for `actor_type: 'Integration'` is the **app id** (not the installation id) — but i have not verified which the api wants. verify in the research phase against the api before the example is canonized.

each tagged **[answered]** (settled now), **[research]** (answer in research phase), or **[wisher]** (needs wisher input).

## what is awkward?

1. **the name→id mismatch is the one forced seam.** every extant github resource here is addressable by its natural key directly in the api url. rulesets aren't — get/update/delete all need the integer `id`. so the idempotent `set` must: list → find by name → (create if absent | update by id). it works cleanly, but it's the first list+filter-by-name lookup in the repo, so it deviates slightly from the "just hit the endpoint with the unique key" rhythm.

2. **repo vs org duplication.** two domain objects + two daos + two cast/get/set sets that are ~90% identical, differing only in scope path and the org's extra `repositoryName` condition. the house style favors wet-over-dry and the task says mirror exactly, so duplication is the right call — but it'll feel repetitive to write.

3. **rule/condition modeling breadth.** github's ruleset `rules[]` is an open-ended discriminated union with per-type `parameters`. modeling it faithfully yet ergonomically (without chasing every rule type) is a judgment call — the design fights the user's desire for "just let me declare the rule" vs the api's sprawling surface. v1 scopes to the task-listed rule types.

4. **bypass actor opacity.** declaring `actorId: 123456` is not self-documenting — a reader can't tell it's release-please. resolving app slug → id (like environment reviewers) would be friendlier but is out of v1 scope; the tradeoff between fidelity-to-api and ergonomics feels uncomfortable but defensible.

5. **diff-stability / false-drift is the real risk (under-weighted above).** the vision promises "re-run `plan` → idempotent no-op", but that only holds if the cast projects github's response down to *exactly* the declared shape. github reshapes ruleset responses server-side: it may **reorder `rules[]`**, inject **default `conditions`**, and **enrich `bypass_actors[]`** (e.g. assign details). the extant pattern already defends against this — `castToDeclaredGithubBranchProtection` (`src/domain.operations/branchProtection/castToDeclaredGithubBranchProtection.ts:19-36`) maps only declared fields and drops server extras. but branch-protection's nested shapes are mostly scalars + a `contexts` **string** array; rulesets introduce **ordered arrays of objects** (`rules`, `bypassActors`, condition `include`/`exclude`). so the ruleset cast must do more than branch-protection's: it must **sort arrays to a canonical order** (e.g. rules by `type`, bypass actors by `actorId`) so a declared `[creation, deletion]` doesn't diff against an actual `[deletion, creation]`. if the cast skips this, every `plan` shows spurious drift and the headline idempotency promise breaks. this is a build-time design requirement, not a nicety — flag for the criteria/blueprint stage. **open**: do we sort to a canonical order in the cast, or require consumers to declare in github's canonical order?
