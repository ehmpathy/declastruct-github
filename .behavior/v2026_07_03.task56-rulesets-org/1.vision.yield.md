# 1.vision — github rulesets as code (org level)

## the wish

> we shipped this for repo level. now lets ship this for org level.
> — [radio task 56](https://github.com/ehmpathy/declastruct-github/issues/56)

repo-level rulesets shipped in [#57](https://github.com/ehmpathy/declastruct-github/pull/57) (`DeclaredGithubRepoRuleset` + dao + get/set/del + cast + tests + dogfood). that delivery deferred the **org-scoped** ruleset to a fast-follow (prior vision Q5). this is the fast-follow: add `DeclaredGithubOrgRuleset` so consumers declare **org-wide** rulesets as code, mirror-matched to the repo-level shape.

---

## the outcome world

### day-in-the-life

vlad codified the aws-oidc "prod apply only from a version tag cut from main" guarantee at the repo level: a `DeclaredGithubRepoRuleset` restricts `v*` tag creation to the rhelease app, per repo. it works — but he has ~30 repos in `ehmpathy`, and each one needs the same tag protection. today that means 30 near-identical `DeclaredGithubRepoRuleset` blocks, and every new repo silently ships **unprotected** until someone remembers to add the block.

after this wish, he declares **one** `DeclaredGithubOrgRuleset` in `provision/github.org/resources.ts` with `conditions.repositoryNameInclude: ['~ALL']`. one resource protects `v*` tags across the whole org — even repos made tomorrow. `declastruct plan` shows a single `CREATE`, `apply` provisions it once, and the org-wide guarantee is codified, diffable, and review-gated.

### before / after

| | before (repo-only) | after (org-level) |
|---|---|---|
| protect `v*` tags org-wide | one `DeclaredGithubRepoRuleset` per repo (~30 blocks) | one `DeclaredGithubOrgRuleset` with `repositoryName: ~ALL` |
| new repo inherits policy | no — must remember to add a block | yes — `~ALL` covers it automatically |
| target a subset of repos | copy the block to each | `repositoryNameInclude: ['svc-*']` |
| review a policy change | 30 diffs | 1 diff |
| drift detection | per repo | one `plan` covers the org rule |

### the "aha" moment

the value clicks when `declastruct plan` prints a single `CREATE DeclaredGithubOrgRuleset` whose `conditions.repositoryNameInclude` is `['~ALL']` — and the reader sees this one line replace a per-repo block on every repo in the org, present and future. org rulesets turn "remember to protect each repo" into "the org protects them by default."

## user experience

### usecases & goals

1. **org-wide tag protection (primary)** — restrict `v*` tag creation to a release app across all (or a glob-selected subset of) repos in the org. the org-scoped completion of the repo-level guarantee.
2. **org-wide branch rulesets** — declare a branch policy (e.g. require linear history, block force-push) once for the whole org instead of per repo.
3. **scoped rollout** — target a subset via `repositoryNameInclude/Exclude` globs (e.g. `['svc-*']`, exclude `['*-sandbox']`) or `repositoryNameProtected` to block target-repo renames.

### contract inputs & outputs

consumers leverage one new domain object, registered in the provider's `daos` and exported from `src/contract/sdks/index.ts` next to `DeclaredGithubRepoRuleset`:

```ts
import { DeclaredGithubOrgRuleset } from 'declastruct-github';

// org-scoped: restrict v* tag creation to the release app, across ALL repos
const orgTagRuleset = DeclaredGithubOrgRuleset.as({
  org,                                   // RefByUnique<typeof DeclaredGithubOrg> (keyed on login)
  name: 'org-protect-version-tags',      // natural unique key, per org
  target: 'tag',
  enforcement: 'active',
  bypassActors: [
    { actorId: 2472031, actorType: 'Integration', bypassMode: 'always' }, // rhelease app id
  ],
  conditions: {
    refNameInclude: ['refs/tags/v*'],
    refNameExclude: [],
    repositoryNameInclude: ['~ALL'],     // org-only condition (repo ruleset has no such field)
    repositoryNameExclude: [],
    repositoryNameProtected: false,
  },
  rules: [{ type: 'creation' }],
});
```

the **only** shape delta from `DeclaredGithubRepoRuleset` is:
- `repo` → `org` (`RefByUnique<typeof DeclaredGithubOrg>`, keyed on `login`)
- `unique = ['repo','name']` → `unique = ['org','name']`
- `conditions` gains `repositoryName*` fields (org-only; the api pairs `repository_name` + `ref_name` for branch/tag org rulesets)

all else — `bypassActors[]`, `rules[]`, `target`, `enforcement`, `id` as `@metadata` primary — is identical.

outputs mirror repo rulesets exactly: `get` returns `HasMetadata<DeclaredGithubOrgRuleset> | null` (`id` populated), `set.findsert`/`set.upsert` return the persisted resource after read-after-write, `set.delete` is idempotent.

### timeline of use

1. add `DeclaredGithubOrgRuleset` to `provision/github.org/resources.ts` (dogfood) and/or a consumer's own org provision
2. `declastruct plan` → review the `CREATE`/`UPDATE`/`no-op` diff
3. `declastruct apply` → ruleset provisioned via octokit (`github.repos.createOrgRuleset`)
4. re-run `plan` → idempotent no-op proves convergence

## mental model

> "same as the repo ruleset, but scoped to the whole org — i declare it once, pick which repos it hits with a glob, and it protects new repos automatically."

| analogy | fit |
|---|---|
| terraform resource | declare desired state, idempotent converge by name |
| the repo ruleset we already ship | identical shape, one scope up |
| an org-wide policy / blanket rule | `repositoryName: ~ALL` = applies to every repo, current and future |

**terms users use vs terms we use:**

| user says | we model as |
|---|---|
| "the org-wide v* tag rule" | `DeclaredGithubOrgRuleset` with a `creation` rule + `refName` condition |
| "apply it to all repos" | `conditions.repositoryNameInclude: ['~ALL']` |
| "only the service repos" | `repositoryNameInclude: ['svc-*']` |
| "let rhelease through" | `bypassActors: [{ actorType: 'Integration', actorId: <app id>, ... }]` |
| "ruleset id" / "ruleset name" | `id` (`@metadata` primary) / `name` (unique per org) |

## evaluation

### how well it solves the goals

- ✅ completes the org half of the rulesets story (repo shipped in #57, org here)
- ✅ one resource protects the whole org — even repos made after the rule
- ✅ slots into the extant declastruct lifecycle with zero new consumer concepts beyond `repositoryName` conditions

### pros

- **~90% mirror of a just-shipped, house-blessed pattern** (`DeclaredGithubRepoRuleset` and its dao/ops/cast/tests) — lowest possible novelty risk
- octokit ships every needed method — **verified installed**: `github.repos.{getOrgRuleset, getOrgRulesets, createOrgRuleset, updateOrgRuleset, deleteOrgRuleset}` (under the `repos` namespace); endpoints `GET/POST /orgs/{org}/rulesets` + `/{ruleset_id}`
- the name→id list+filter lookup, canonical-sort cast, and get-before-write idempotency were all solved for repo rulesets — org reuses the exact idioms

### cons / tensions

- **CI cannot run the org lifecycle.** org rulesets require `admin:org` scope, which CI/CD tokens lack (house brief `.agent/repo=.this/role=any/briefs/howto.test-admin-resources.md`). the repo ruleset's integration/acceptance ran green in CI with the test-auth app; the org ruleset **cannot**. this is the single biggest difference from #57 and shapes the whole test strategy (below).
- **repo↔org duplication.** a second domain object + dao + cast/get/set/del set that is ~90% identical to the repo version. house style favors wet-over-dry and the task says mirror exactly, so duplication is the right call — but it repeats work.
- **`repositoryName` condition adds cast/diff surface.** it's another ordered-array condition that must sort to a canonical order in the cast (like `refName`), or `plan` shows spurious drift.

### edgecases & pit-of-success

| edgecase | how the contract keeps users safe |
|---|---|
| org ruleset absent yet | `get.byUnique` (list `getOrgRulesets` + filter by name) returns `null` → `set` creates (findsert/upsert symmetry) |
| duplicate name in org | list+filter expects exactly one → fail-fast (`UnexpectedCodePathError`) if >1 (same safety net as repo) |
| `get.byPrimary` (by id) | `undefined` — the id doesn't carry the org (login) the endpoint needs; lookup goes through `byUnique` (mirrors repo dao) |
| repo-only vs org-only conditions | separate domain objects: `repositoryName*` exists **only** on `DeclaredGithubOrgRuleset`; the type system forbids it on repo |
| 404 on get/delete | return `null` / no-op — idempotent, matches extant pattern |
| new repo made after the rule | `~ALL` include covers it automatically — the headline org benefit |

## open questions & assumptions

### assumptions made (inherited from #57, re-affirmed)

1. **mirror `DeclaredGithubRepoRuleset` exactly** — file layout (`src/domain.operations/orgRuleset/`), cast fn, `ContextGithubApi & ContextLogTrail`, `asProcedure`, one-export-per-file, fail-fast, read-after-write, canonical-sort cast. lowest-risk path and matches the task.
2. **separate domain object**, not one shared repo/org object with optional `org`/`repo` — keeps repo vs org type-safe, no adhoc optionality (rule.forbid.adhoc-refs). org adds `repositoryName*` conditions.
3. **`primary = ['id']`, `unique = ['org','name']`** — org ref is `RefByUnique<typeof DeclaredGithubOrg>` keyed on `login`.
4. **bypass actor ids passed as-is** (no app-slug → id resolution in v1), same as repo.
5. **`bypassActors`, `conditions`, `rules` as nested `DomainLiteral`** inline shapes — same as repo ruleset (flat fields only, per the domain-objects serialize constraint noted in `DeclaredGithubRepoRuleset.ts:105-109`).
6. **`conditions` flattened** — `refNameInclude/refNameExclude` + `repositoryNameInclude/repositoryNameExclude/repositoryNameProtected`, all flat on one `DomainLiteral` (no nested `ref_name`/`repository_name` objects), to match the repo cast's flatten rationale.

### questions — resolved by the wisher (2026-07-03)

the wisher confirmed all three proposals and confirmed dogfood is wanted:

1. **`target` scope for org — include `'push'`?** — **[answered]** keep symmetric to repo: `target: 'branch' | 'tag'` for v1. `'push'` deferred (cheap to add later if a usecase surfaces).
2. **condition breadth — `repositoryId` / `repositoryProperty`?** — **[answered]** v1 covers **`repositoryName`** only (globs + `~ALL` + `protected`). `repositoryId`/`repositoryProperty` out of scope.
3. **dogfood — apply a real org ruleset?** — **[answered]** yes, dogfood it. scope stays **narrow + non-enforced** to avoid org-wide blast radius: `enforcement: 'evaluate'` + `repositoryNameInclude: ['declastruct-github-demo']` (not `~ALL`). the `~ALL` power stays documented but unexercised against prod. the org lifecycle still cannot run in CI (`admin:org`), so dogfood + `TEST_ORG_ADMIN`-gated integration + acceptance plan snapshots are the proof, per the repo precedent.

### must research at build time — **[research]**

- exact octokit **response/parameter field names** for org `conditions.repository_name` (confirmed shape: `{ include?, exclude?, protected? }`) and how `getOrgRulesets` list-response projects `name`/`id` — verify against `@octokit/types` when the cast is built (repo cast is the template).
- whether `createOrgRuleset` **requires** the `repository_name` condition for `target: 'tag'` (the openapi doc says branch/tag org conditions "should contain both `repository_name` and `ref_name`") — verify the api rejects a tag ruleset that omits `repository_name`, to decide if `repositoryName*` is required (non-null) or optional in the domain object.

## groundwork

### external research

- **octokit org-ruleset methods exist and are installed** — verified in `node_modules/.pnpm/@octokit+plugin-rest-endpoint-methods@13.5.0_.../method-types.d.ts`: `createOrgRuleset` (9803), `deleteOrgRuleset` (10058), `getOrgRuleset` (10746), `getOrgRulesets` (10756), `updateOrgRuleset` (11815). **note**: they live under the **`repos`** namespace (`github.repos.getOrgRuleset`), not `orgs` — same as the repo ruleset methods.
- **endpoint paths confirmed** in `parameters-and-response-types.d.ts:2846-2849`: `POST /orgs/{org}/rulesets` (+ list/get/update/delete by `{ruleset_id}`). matches the task's api reference.
- **org conditions shape confirmed** in `@octokit/openapi-types@24.2.0/types.d.ts:17994-18049`: org rulesets add `repository_name: { include?, exclude?, protected? }` (also `repository_id`, `repository_property`). the doc states branch/tag org conditions "should contain both `repository_name` and `ref_name`"; push rulesets don't need `ref_name`. → drives the `repositoryName*` fields + build-time research item 2 above.
- **key constraint (unchanged from repo)**: get-by-id needs the server-assigned `ruleset_id`; there is no get-by-name endpoint → by-name lookup = `getOrgRulesets` (list) + filter on `name`.

### internal research

- **`DeclaredGithubRepoRuleset`** (`src/domain.objects/DeclaredGithubRepoRuleset.ts:1-116`) — the exact template: `primary=['id']`, `unique=['repo','name']`, nested `RefByUnique` + `DomainLiteral` (bypassActors/conditions/rules); flat condition fields per the serialize constraint (lines 105-109). org version swaps `repo`→`org`, `unique`→`['org','name']`, adds `repositoryName*` to conditions.

- **`DeclaredGithubOrg`** (`src/domain.objects/DeclaredGithubOrg.ts:68-79`): `unique=['login']`, `primary=['id']` → org ref is `RefByUnique<typeof DeclaredGithubOrg>` on `login`. matches how `DeclaredGithubOrgSecret`/`DeclaredGithubTeam` reference the org (`{ login: 'ehmpathy' }`).

## what is awkward?

1. **the CI test asymmetry is the real tension.** #57 shipped with integration + acceptance green in CI (repo scope, test-auth app). the org ruleset **cannot** run its lifecycle in CI — `admin:org` is out of reach. so its confidence story leans on unit-tested cast + `TEST_ORG_ADMIN`-gated integration + dogfood apply, which is weaker/less automated than the repo ruleset's. precedented (teams do it) and house-sanctioned, but it means "org ruleset works" is proven by a human via dogfood, not by CI. the mechanic house rules forbid failhide/skip **as a shortcut** — here the skip is the documented, wisher-known admin-scope exception, not a hidden dodge. the criteria/blueprint stage must encode this precisely so it does not read as a failhide.

2. **repo↔org near-duplication.** a second ~90% copy feels repetitive and invites "why not generalize?" — but the repo/org api paths, condition shapes, and type-safety needs differ enough that wet-over-dry + mirror-exactly is the right, house-aligned call. worth a note up front so it is not second-guessed in review.

3. **dogfood blast radius.** unlike a repo ruleset on a demo repo, an org ruleset on `ehmpathy` can touch every repo. `~ALL` in a dogfood is genuinely dangerous. the awkwardness: the *headline feature* (`~ALL` across the org) is exactly the capability we should **not** dogfood at full scope — so the dogfood proves the mechanism on a narrow, non-enforced scope, and the `~ALL` power stays documented but unexercised against prod. that gap between "what we demo" and "what it can do" is a little uncomfortable but is the safe choice (wisher confirms via open question 3).

4. **`repositoryName` required-vs-optional is unresolved.** the openapi hint that tag org rulesets "should contain both `repository_name` and `ref_name`" suggests `repositoryName*` may be **required** by the api for tag/branch targets — but "should" is not "must". a required (non-null) model is safer for the pit-of-success but needs the build-time api check (research item 2). an optional model risks an opaque api rejection at `apply` time. a small but real modeling fork the vision cannot fully close without a call to the api.
