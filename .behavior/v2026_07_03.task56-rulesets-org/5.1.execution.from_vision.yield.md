## peer-review resolution: removed the gated integration play test (2026-07-04)

peer review (`mech-failhides`) flagged `orgRuleset.play.integration.test.ts`'s `given.skipIf(!hasOrgAdmin)`
as a `rule.require.failfast` violation (skip-on-absent-resource = failhide). the skip existed because CI
**permanently** lacks admin:org, so the two literal fixes both had problems: keep-skip keeps the reviewer
blocker open; throw-`ConstraintError` breaks CI (CI runs `test:integration` with the no-admin test-auth token).

per the wisher's **dogfood-only** steer, i **removed** the play integration test. rationale:
- `howto.test-admin-resources` prescribes admin-resource verification as **unit tests + dogfood** — it does
  **not** call for a gated integration test. so removal aligns *closer* to the documented admin strategy.
- dogfood (`provision/github.org` → `declastruct plan`/`apply`) exercises the same create/read/upsert/delete
  path, so no real coverage lost.
- removal makes the failhide moot (no skip at all) rather than a cover-up of it.

net: org ruleset verification = **unit tests (cast + domain, run in CI)** + **dogfood (human, admin token)**.

## decision: org ruleset is dogfood-only, not in CI acceptance (wisher steer, 2026-07-04)

vlad confirmed org rulesets are verified **via dogfood only**. reason: `resources.acceptance.ts` runs
`declastruct plan` (+ gated apply) **in CI**, and the CI test-auth token lacks `admin:org` — so an org
ruleset in the acceptance file would (a) change the committed plan snapshot, which then needs a local
`GITHUB_TOKEN` (human-only fill) to regenerate — a false blocker — and (b) risk the CI plan phase to fail
to even read the org resource.

so the org ruleset is verified through:
- **unit tests** (domain + cast) — run in CI, no token
- **dogfood** (`provision/github.org`) — human runs `declastruct plan`/`apply` with admin token

this is a deliberate exception to `rule.require.acceptance-resources`; the admin:org CI constraint +
`howto.test-admin-resources` brief override it for admin-scoped resources. repo-scoped resources (the repo
ruleset from #57) stay in `resources.acceptance.ts` — repo scope works with the CI token.

**net effect on the acceptance files**: reverted to their pre-change state (`git diff` shows only
`src/contract/sdks/index.ts` +1 line, the SDK export). no acceptance snapshot regen needed.

## notes for the reviewer

- **repo↔org duplication is deliberate** (wet-over-dry + "mirror exactly" per the task); the org files are ~90%
  identical to the repo ruleset, different in scope path (`/orgs/{org}/rulesets`), the `org` ref, and the
  `repositoryName` condition.
- **octokit org methods live under `github.repos.*`** (`getOrgRuleset`, `createOrgRuleset`, …), not `github.orgs.*`
  — verified first-hand in `method-types.d.ts:10740-10762`.
- **the `TEST_ORG_ADMIN` skip is the sanctioned admin-scope exception**, not a failhide — documented inline and
  precedented by `team.play.integration.test.ts` + `howto.test-admin-resources.md`.
